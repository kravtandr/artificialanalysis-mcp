# AGENTS.md — правила разработки artificialanalysis-mcp

MCP-сервер на TypeScript для подбора AI-моделей через Artificial Analysis Data API v2.
Полное ТЗ — в `SPEC.md`; при противоречии SPEC.md главнее. Спека API —
`artificial-analysis-openapi.yaml` (источник истины по эндпоинтам и полям).

## Команды

```bash
npm ci                  # установка строго по lock-файлу
npm run dev             # запуск в watch-режиме (stdio)
npm run build           # сборка в dist/
npm run lint            # ESLint + Prettier check
npm run typecheck       # tsc --noEmit
npm test                # vitest
npm run smoke           # MCP-handshake против собранного dist/
npm run generate:types  # регенерация src/api/types.gen.ts из OpenAPI
```

Перед коммитом обязательно зелёные: `lint`, `typecheck`, `test`, `build`.

## Архитектурные инварианты

- **Слои не смешивать**: `tools/*` не делают HTTP-запросов напрямую — только
  через `catalog.ts` → `api/client.ts`. Фильтрация и fuzzy-поиск — только в
  `match.ts`. Форматирование ответов — только в `format.ts`.
- **stdout священен**: в stdio-режиме stdout принадлежит протоколу MCP.
  Любой лог — только в stderr. Никаких `console.log` в коде сервера.
- **Квота — дефицитный ресурс** (Free: 100 запросов/день). Любая новая фича
  обязана работать через кэш `cache.ts`. Прямые «свежие» запросы в обход кэша —
  только с обоснованием в PR.
- **Кэш не удаляет протухшее**: записи по истечении TTL помечаются stale, но
  остаются — при `429`/`5xx` отдаётся stale-копия с пометкой возраста, если она
  есть; без stale-копии ошибка пробрасывается инструменту (SPEC.md §3.3).
  Обычный Map с expiry-удалением это требование молча ломает.
- **Один промах — один запрос**: параллельные промахи по одному ключу кэша
  коалесцируются (в кэше живёт промис загрузки). Cache stampede при квоте
  100/день недопустим.
- **Синглтоны уровня процесса**: `AAClient`, кэш и каталог создаются один раз
  при старте. В stateless-HTTP-режиме SDK создаёт server/transport на каждый
  POST — состояние в них не хранить.
- **Каждый fetch — с таймаутом** (`AbortSignal.timeout`, SPEC.md §3.2):
  зависший запрос к AA вешает вызов инструмента у клиента.
- **`null` ≠ 0**: null в данных AA означает «не измерено». Числовые фильтры
  не пропускают записи с null; в сортировках null уходит в конец.
- **Tier-aware, не tier-hardcoded**: не завязываться на Free-тариф; выбор
  эндпоинта (`/models` vs `/models/free`) делает `AAClient` по фактическому тарифу.
- **Атрибуция обязательна**: каждый ответ инструмента содержит
  `Source: Artificial Analysis (https://artificialanalysis.ai)` — требование лицензии API.

## Стиль кода

- TypeScript strict, ESM, Node ≥ 20. `any` запрещён; на границах (входы
  инструментов, ответы API) — zod-схемы, дальше только выведенные типы.
- `src/api/types.gen.ts` руками не править — только `npm run generate:types`.
  Обновили `artificial-analysis-openapi.yaml` → сразу регенерируйте типы и
  закоммитьте вместе (CI проверяет синхронность).
- У каждого поля входной схемы инструмента — `.describe()`: эти описания
  читает LLM-клиент, пишите их как документацию для модели (по-английски).
- Комментарии — только о неочевидных ограничениях (семантика null, лимиты
  тарифов), не о том, «что делает следующая строка».
- Именование файлов — kebab-case; один инструмент = один файл в `src/tools/`.

## Тесты

- Реальный API в тестах **не вызывается никогда** — только undici MockAgent
  и фикстуры из `tests/fixtures/` (взяты из примеров OpenAPI-спеки).
- Новый инструмент = интеграционный тест через `InMemoryTransport` +
  unit-тесты чистой логики. Обязательно покрывать: happy path, пустой
  результат, ошибки API (401/403/429), tier-фолбэк.
- Багфикс начинается с падающего теста, воспроизводящего баг.

## Секреты и безопасность

- `ARTIFICIAL_ANALYSIS_API_KEY` живёт только в env (`.env` в `.gitignore`,
  образец — `.env.example` без значений). Ключ не логировать, не включать
  в сообщения об ошибках, не отдавать в ответах инструментов.
- Никогда не коммитить записанные ответы API, содержащие заголовки запроса.

## Git и релизы

- Conventional Commits: `feat:`, `fix:`, `docs:`, `test:`, `chore:`, `ci:`,
  `refactor:`. Сообщение отвечает «зачем», а не «что».
- Работа через ветки + PR в `main`; прямые push в `main` — только для
  тривиальных правок документации.
- Релиз: поднять `version` в `package.json` → коммит → тег `vX.Y.Z` → push
  тега. `release.yml` сам опубликует npm-пакет и образ в GHCR. Тег обязан
  совпадать с версией пакета (CI это проверяет).
- SemVer; до 1.0.0 ломающие изменения интерфейса инструментов допустимы
  в минорных версиях, но фиксируются в release notes.

## Чего не делать

- Не добавлять зависимости без необходимости: fetch встроен, HTTP-фреймворк
  для одного эндпоинта не нужен. Каждая новая зависимость обосновывается в PR.
- Не реализовывать `/api/v2/critpt/evaluate` — вне охвата (см. SPEC.md §9).
- Не превращать инструменты в прокси сырых JSON-ответов API: ответы должны
  быть компактными и отобранными — их читает LLM, токены дороги.
